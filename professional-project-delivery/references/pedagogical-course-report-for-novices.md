# Pedagogical course-style reports for novice users

Use this reference when Moufadal asks for a report that should teach while reporting, especially after complex VPS/agent/code work.

## Trigger phrases

- “rapport sous forme d’un cours”
- “je suis perdu”
- “explique-moi en même temps que j’apprends”
- “pas de code / pas de gros pavé incompréhensible”
- “diagrammes / illustrations”

## Report contract

A course-style technical report is not a raw engineering log. It must separate:

1. **Learning path** — what the user should understand first, second, third.
2. **Operational truth** — what was actually verified, what remains open.
3. **Evidence** — paths/logs/tests/screenshots, but not dumped inline.
4. **Actionability** — next step, restart command, rollback command.

## Recommended structure

1. **Hook / TL;DR** — one simple sentence explaining the whole system.
2. **Mental model** — analogy first, e.g. “Hermes = chef d’orchestre”.
3. **Diagram** — request → classify → delegate → artifact → QA → Telegram → reprise.
4. **Glossary** — explain terms like artifact, log, QA, observe/warn/block.
5. **Step-by-step walkthrough** — what happened, in chronological order.
6. **Proof cards** — one card per criterion with status PASS/PARTIAL/FAIL and evidence path.
7. **Risk lesson** — why a tempting shortcut is dangerous, e.g. hard `block` too early.
8. **Practice / self-check** — 3–5 questions or “can you explain this back?” prompts.
9. **Reprise / rollback** — exact path and command, clearly separated.

## Design principles

- Prefer editorial HTML with cards, flow diagrams, callouts, and short sections.
- Use diagrams only when they explain a process or relationship; avoid decorative visuals.
- Keep code blocks out of the main reading path. Use short inline commands only when necessary.
- Put raw logs and JSON in artifact files, not in Telegram or the beginner report body.
- Use status labels the user can understand: `Terminé`, `Partiel`, `Bloqué`, `À surveiller`.
- If a criterion is not fully true, say so plainly; do not convert “warn-ready” into “production-ready”.

## QA for the report itself

Before delivering:

- Open or parse the HTML artifact and verify it contains the promised sections.
- Check that the report links/paths to the real QA artifacts exist or are at least generated by the same run.
- Ensure the first screen is understandable without scrolling into raw logs.
- Ensure every “PASS” claim maps to an actual proof path or command output.
