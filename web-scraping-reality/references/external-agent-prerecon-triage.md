# External agent / ChatGPT / Comet scraping reports — triage protocol

## When to use

When the user provides a PDF, note, report, screenshot, or browser-agent output generated by another AI about how to scrape a site.

## Rule

Treat the report as **pre-recon**, not as ground truth or a production spec. It can accelerate discovery, but every endpoint, protection claim, pagination rule, and technology assertion needs a local artifact before it is given to implementers.

## Fast workflow

1. Extract the document text locally.
2. Split claims into:
   - confirmed facts;
   - plausible but unverified hypotheses;
   - suspicious/hallucinated details;
   - missing production requirements.
3. Run quick live probes for the high-value claims:
   - root page status/headers;
   - sitemap/robots;
   - sample catalogue/viewer URLs;
   - candidate API/asset endpoints;
   - one known asset download.
4. Compare against existing local artefacts/skills/session notes. Local artefacts beat AI-generated reports.
5. Rewrite the report into a **capteur brief**:
   - what to call;
   - what fields to store;
   - how to paginate;
   - how to validate completeness;
   - what is still unknown.
6. Label guessed endpoints explicitly as hypotheses; never hand guessed endpoints to implementers as facts.

## Red flags in AI-generated scraping reports

- Endpoint names such as `/api/prospectus`, `/v2/catalogues/<uuid>`, `asset_id`, `pages_count` are presented without raw response evidence.
- A server stack is inferred from one layer while the live scraper actually sees Cloudflare or another CDN.
- The report explains download URLs but omits completeness QA.
- It says “no protection” without testing rate limits, retries, and representative recent/large items.
- It describes one viewer family but ignores other viewer types on the same site.

## Output format for the user

Keep the response practical:

- Verdict: usable / partially useful / not reliable.
- Verified live facts with artefact paths or command outputs.
- Useful parts to reuse.
- Incorrect/unsupported claims.
- Missing items required before coding.
- A rewritten implementer brief if useful.

## Lapub.re lessons from 2026-06-12

A ChatGPT-agent PDF correctly identified Next.js, sitemap discovery, public flipbook assets, and direct-image downloads. It was still incomplete as a production guide because it underplayed:

- MAGZ / `interactive-catalogue.com` JSON pagination where `pagination=1` can return only the first batch while `detail.totalPages` is much larger.
- FLIPPING V2 Unicode URL corruption risks if decoding the whole HTML with `unicode_escape`.
- Strict QA: expected pages = DB pages = local valid files, with retries and manifests.
- The need to distinguish inferred API endpoints from locally captured endpoints.

For LapubRe, give implementers the AI report only after hardening it with these field-proven additions.
