# VPS disk cleanup audit pattern — 2026-06-30

Use this as a concrete pattern when Moufadal asks for a “gros ménage” or disk-space cleanup but has **not** authorized exact destructive targets.

## Situation

A scheduled 04:00 Réunion job had to audit VPS disk usage, free safe space, and produce Markdown + HTML reports under `/opt/data/artifacts/vps-disk-cleanup-20260630/`.

The important safety constraint: “gros ménage” is **not** permission to delete broad `/opt/data` trees, projects, Docker volumes/images, DBs, credentials, backups, git history, or prod logs.

## Safe sequence that worked

1. Create a dated artifact directory with `raw/` and `qa/`.
2. Capture before-state evidence:
   - `df -hT / /opt /opt/data /tmp /var/tmp`
   - `du -xhd1 /`, `/opt`, `/opt/data`, `/opt/hermes` where readable
   - `docker system df -v` and `docker ps`
   - `journalctl --disk-usage` if available
   - bounded top files >100M over relevant roots
   - `/tmp`, caches, project artifacts, node_modules/venvs inventory
3. Write `CLEANUP_PLAN.md` before deletion with Green/Yellow/Red classification.
4. Delete only exact, inspected, rebuildable temp/cache paths. In this session:
   - old generated `/tmp/immo-*stage*` directories;
   - old `/tmp/immo_clean_v3_test`;
   - selected old `/tmp/pycache-*` and Hermes QA pycache dirs;
   - official cache commands: `uv cache prune`, `pip cache purge`.
5. Do **not** prune Docker blindly. A `<none>` image can be used by active containers (`hermes-gateway`, `hermes-dashboard`). Treat Docker prune as Red unless the exact image/container/volume is identified and backed up.
6. Do **not** delete large project artifacts just because they are large. Classify them as validation candidates with size, reason, and a proposed retention policy.
7. Capture after-state evidence:
   - `df -hT` after;
   - `du` after for touched areas;
   - `docker ps` after;
   - lightweight health checks for key public/local services.
8. Produce durable reports:
   - `REPORT.md` for full findings;
   - `REPORT.html` mobile-readable;
   - raw logs under `raw/`;
   - `qa/verify_report.json` checking file existence, size, HTML parse, and required tokens.

## Reporting lesson

If safe cleanup succeeds but high-value deletions remain unapproved, the verdict must be:

> Partiellement terminé — audit fait, ménage sûr effectué, suppression destructive bloquée par validation.

Do not claim a full cleanup just because some space was freed.

## Candidate classification examples

- Critical / do not touch without validation:
  - active Docker images/volumes/containers;
  - DB files such as `/opt/data/state.db`;
  - `.env`, secrets, credentials, git history, backups.
- Important validation candidates:
  - large project artifact directories;
  - dated tarballs/snapshots where a retention policy can be agreed;
  - browser/model caches that may be operationally useful.
- Safe/Yellow when exact and inspected:
  - old generated temp directories under `/tmp`;
  - rebuildable tool caches via official prune commands;
  - old `__pycache__`/temporary QA directories outside active projects.

## Useful final Telegram framing

- State before/after disk usage and estimated freed space.
- Separate “actions done” from “candidates not deleted”.
- Include `MEDIA:/absolute/path/REPORT.html` when an HTML report exists.
- Give rollback honestly: caches/temp are regenerated by rerunning producing QA/build; no service rollback if no service/config changed.
